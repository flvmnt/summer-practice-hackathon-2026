## Brief reconciliation

The brief states "commit f471fb3 dropped Romanian from served locales" and asks about "+22 to ro/common.json in current diff". Both are stale:

- `git status` is clean except for two untracked audit partials. There is no in-progress diff to `messages/ro/common.json` or `src/i18n/routing.ts`.
- `f471fb3` set `locales: ["en"]` then `cd41e08` ("translate landing-page side card + live cards to RO") REVERTED to `["ro","en"]` with `defaultLocale: "ro"` because "the earlier locale removal turned out to lose the multi-language rubric points the user wanted to keep."
- The most recent i18n-touching commit is `250d822 fix(i18n): close locale leaks across core flows` (HEAD~3). Its `messages/ro/common.json` change was **+22 / -1 lines**, almost certainly the "+22" the brief refers to.

So the audit treats HEAD as the source of truth: **both `ro` and `en` are served, `/ro` is the default**, and the hackathon is still on the multi-language bonus track per spec `docs/specs/16-i18n-plan.md:12` and `docs/specs/13-scoring-coverage.md:81`.

## Findings

### P0 — User-visible broken render

**P0-1. `today.queued.*`, `today.confirmed.*`, `today.saidNo.subhead` — multiple keys never added to either catalog; component falls back to hardcoded English even on `/ro/today`.**

`src/components/today/TodayPromptCard.tsx:325-405` reads (e.g.) `copy.queued?.subhead`, `copy.queued?.lookingFor`, `copy.queued?.elapsedJustNow`, `copy.queued?.elapsedSeconds`, `copy.queued?.elapsedMinutes`, `copy.queued?.progressNearby/Skill/Final`, `copy.queued?.planBHint`, `copy.confirmed?.headline`, `copy.confirmed?.venueLine`, `copy.confirmed?.rosterLine`, `copy.confirmed?.calendarPending`, `copy.saidNo?.subhead`, `copy.saidNo?.browseHint`. The page passes `t.raw("card")` (`src/app/[locale]/today/page.tsx:206`) so the actual map handed in is `messages/en/common.json` `today.card` (or RO equivalent). Inspecting both catalogs:

```text
EN today.card.queued    = { title, searchingLabel, planB, tryRun, createSmall }
EN today.card.confirmed = { label, startsLabel, openChat, calendar }
EN today.card.saidNo    = { title, change }
```

None of `subhead`, `lookingFor`, `elapsed*`, `progress*`, `planBHint`, `headline`, `venueLine`, `rosterLine`, `calendarPending`, `browseHint` exist in EN OR RO. Result at runtime when locale is `ro`:

- "While you wait, try one of these:" (`TodayPromptCard.tsx:340`) - shipped in RO
- "Looking for {sport}" prefix (`:333`) - English template wrapping a RO sport name -> "Looking for Fotbal"
- "Searching" (`:336`)
- "Plan B" (`:337`)
- `copy.found?.in ?? "You're in · ${sportLabel}"` actually does have `today.card.found.in = "You're in"` so this one works on EN; on RO `today.card.found.in = "Ești înăuntru"` - OK.
- `copy.confirmed?.headline.replace(...) ?? "${sportLabel} · ${startTime}"` (`:387`) - pure punctuation fallback, OK
- `copy.confirmed?.rosterLine.replace(...) ?? "8 going · 2 maybe"` (`:392`) - English ships when key absent, which it is

User on `/ro/today` who lands in `state === "queued"` or `state === "said-no"` sees a mix of Romanian and bare English. This is the exact bug the `fix(i18n): close locale leaks` commit was claimed to close - but the leaks remain, just relocated to keys that don't exist.

`lint:i18n` passes only because `scripts/check-i18n.mjs:74` walks namespaces from disk, not from component usage; it cannot detect references to never-written keys.

Fix scope: add `today.card.queued.{subhead, lookingFor, elapsedJustNow, elapsedSeconds, elapsedMinutes, progressNearby, progressSkill, progressFinal, planBHint}`, `today.card.confirmed.{headline, venueLine, rosterLine, calendarPending}`, `today.card.saidNo.{subhead, browseHint}` to BOTH `messages/en/common.json` and `messages/ro/common.json`.

### P1 — Promise / parity gap

**P1-1. Missing `/today` language switch — multi-language bonus row claim is not provable.**

Spec `docs/specs/16-i18n-plan.md:12`: "Show a language switch on `/today` for demo proof before claiming the multi-language bonus row." Spec `docs/specs/06-ui-flows.md:724` step 14: "Switch language RO/EN." Spec `docs/specs/08-bonus-features.md:102`: "switch language on `/today`". And `src/lib/demo/scoring-proofs.ts:339-343` claims the bonus with `evidenceLabel: "switch RO/EN"`.

Searched: `LocaleSwitcher`, `LanguageSwitch`, `setLocale`, hard-coded `/en` ↔ `/ro` toggle in `src/components/layout/`, `src/components/today/`, anywhere under `src/app/[locale]/today/`. **No locale switcher component exists.** Catalog has `home.languageSwitch = "RO"` (`messages/en/common.json:7` area) but it's never imported (`grep` returned zero `t("home.languageSwitch")` and zero `t("languageSwitch")`).

Practical consequence: a judge on `/ro/today` cannot reach `/en/today` through the UI. The bonus row in `scoring-proofs.ts` describes proof that doesn't exist.

**P1-3. `cd41e08` deliberately introduced inline-locale switching for landing live cards instead of message keys.**

This is honest but it means another ~12 strings ("Live - today", "Football today", etc.) are in `.tsx`, not in `messages/`. Same maintenance liability as P1-2.

**P1-4. `home.languageSwitch = "RO"` key is dead.**

`messages/en/common.json` (and ro mirror) define `home.languageSwitch` but no `t("home.languageSwitch")` exists in the codebase. It's pre-wired storage for a switcher that was never built. Either delete the keys (cleanup) or build the switcher (which P1-1 demands anyway).

### P2 — Cleanup

**P2-1. `signup`, `i`, `leaderboard` pages still use `locale === "ro" ? RO : EN` ternaries inside JSX.**

62 occurrences of `locale === "ro" ? ... : ...` across `src/`. Worst hotspot: `src/app/[locale]/signup/page.tsx:19-51` (~20 ternaries). Also `src/app/[locale]/i/[token]/page.tsx`, `src/app/[locale]/leaderboard/page.tsx:99,109`. Same antipattern as P1-2 in spirit but smaller surface; the spec calls for next-intl-driven message keys throughout.

`src/app/[locale]/leaderboard/page.tsx:109`: `const headerPlayer = locale === "ro" ? "Jucător" : "Player"` - directly contradicts P0-3 (which uses unconditional English). Two surfaces, two strategies, neither uses the catalog.

**P2-2. AppLocale type kept wider than served locales by design.**

`f471fb3`'s commit message documented intentionally keeping `AppLocale` wider after dropping `ro`, then `cd41e08` re-added `ro` to served locales without consolidating. The current state happens to be consistent (`AppLocale = "ro" | "en"`, both served), but the dead-comment rationale ("40+ files with `locale === "ro" ? RO : EN` branches still typecheck") still shows up in the wider audit (P2-1). When the inline branches finally get migrated, drop those 40+ ternaries in the same sweep.

**P2-3. `messages/en/common.json` `today.card.found` is the only `found.*` populated; matching `today.found.*` doesn't exist.**

Subtle: `today.found.*` was added at `messages/en/common.json:329-334` (per RO diff), but the component reads `copy.found` from `t.raw("card")` which means `today.card.found` - and that one does exist with the same keys. So this is dual-storage, not a missing key. Recommend deleting `today.found`/`today.queued`/`today.saidNo`/`today.confirmed` at the `today.*` (non-card) namespace level since they are not consumed - and lift the in-card object to a flat `today.found`/etc. for consistency.

Verify: `grep -rn 't("today.found' src` and `grep -rn 't("today.queued' src` both return zero hits. Only `t.raw("card")` (the parent) is consumed.

**P2-4. `TodayPromptCard.tsx:218-219` headline fallback is hardcoded `"ShowUpToday?"`.**

`const headline = promptHeadline || prompt.messageText || "ShowUpToday?"` (`:219`). When both fall through, the literal English/Romanian-equivalent string ships. Brand name is OK to keep, but routing through `t("title")` keeps it in one place.

**P2-5. `MobileTabBar.tsx` and `HeaderBell.tsx` are clean.** Both use `useTranslations` with parameter-substituted ICU. No issues. (Spot-checks: `MobileTabBar.tsx:51,74,80`, `HeaderBell.tsx:24-28`.)

**P2-6. `LocationForm.tsx`, `SetupBanner.tsx`, `GroupChatForm.tsx`, `GroupMembersList.tsx`, `GroupTabs.tsx`, `MatchPercentPanel.tsx` are clean.** All accept a `copy` prop populated server-side from `t.raw(...)` with no inline literals. Kudos.

**P2-7. `PhotoForm.tsx` mostly clean but has hardcoded `SPORT_LABELS` table at `:24-35`.**

Also `:139`: `s.source === "ai" ? "from photo" : "fallback hint"` - English literals in client state.

### P3 — Cosmetic

**P3-1. 62 EN/RO message values are byte-identical.**

Most are legitimate (proper nouns, abbreviations, brand names, placeholders): `Padel`, `Yoga`, `Badminton`, `Strava`, `Alex Ionescu`, `XXXX-XXXX-XXXX`, `Timișoara`, `{count}/240`. A few that should NOT be identical and were missed by the translator:
- `home.loopSteps.2` = `"ShowUpToday"` (acceptable - brand)
- `today.title` = `"ShowUpToday"` (acceptable - brand)
- `onboarding.profile.eyebrow` / `onboarding.sports.eyebrow` / `onboarding.location.eyebrow` = `"Onboarding"` (English word in RO; should be `"Configurare cont"` or `"Bun venit"`)
- `event.captainBrief.sourceAi` = `"AI · live"` (acceptable - tech label)
- `event.captainBrief.sourceFallback` = `"Auto · v1"` (acceptable)

ROADMAP-ro-translations.md exists and acknowledges this backlog. Honest fallback, low priority.

**P3-2. `today.queued.searchingLabel`** = `"Searching"` (EN) / `"Caut"` (RO). RO is grammatically a 1st-person verb ("I search"); should be `"Se caută"` (passive, "Searching") for parity with the English present-continuous tone. Cosmetic.

**P3-3. `today.confirmed.startsLabel`** = `"Starts soon"` (EN) / `"Începe curând"` (RO). RO is a fine literal but the EN never gets a `{time}` token despite the component calling `.replace("{time}", startTime)` at `TodayPromptCard.tsx:381`. So both locales render `"Starts soon"` and the time is silently dropped. Could also be a P0 if the time was supposed to be visible - but the component also passes `whenLabel` two lines later which DOES include `{time}`, so this just makes the chip redundant.

**P3-4. Sport-label tables duplicated in 6 catalog locations.**

`messages/en/common.json` lines 222, 366, 515, 689, 976, 1063 all repeat the same `{football, basketball, tennis, ...}` table. Same in RO. Extract to `sports.labels.*` and reference. Lots of cleanup wins for one canonical table.

## Sanity audit

- **Catalog parity (machine-checked):** `messages/en/common.json` and `messages/ro/common.json` both contain exactly **992 keys**. Set diff in both directions: **0**. ICU argument set parity: **0 mismatches** across all 992 strings. `pnpm lint:i18n` passes locally (`i18n check passed.`).
- **`scripts/check-i18n.mjs:14-21` ICU validator** correctly catches mismatched curly braces and argument-name divergence; it will not catch the missing-key-in-component class of bug shown in P0-1.
- **Pluralization (`{count, plural, ...}`):** 3 EN keys, 3 RO keys. EN uses `one/other`; RO correctly uses `one/few/other` (matches Romanian's three-form plural). Examples:
  - `messages/en/common.json:650` `"votes"` -> `"{count, plural, one {# vote} other {# votes}}"`
  - `messages/ro/common.json:650` -> `"{count, plural, one {# vot} few {# voturi} other {# voturi}}"`
  - Same shape for `eventsList.activeCount` (`:1106`) and `eventsList.captainCount` (`:1107`). Looks correct.
- **HTML/XSS:** No `<` / `>` / `&lt;` / `&gt;` characters in either catalog. No `t.rich(` consumers. No risk surface.
- **Locale prefix mode:** `routing.ts:6 localePrefix: "always"` + `routing.ts:7 localeDetection: false`. `proxy.ts:39 matcher` excludes `api`, `_next`, `_vercel`, and any path with a dot. `next.config.ts` wires next-intl via `createNextIntlPlugin("./src/i18n/request.ts")`. `request.ts` uses `hasLocale` for safe fallback to `defaultLocale`. Configuration is consistent and correct.
- **`/api/events/[eventId]/ics/route.ts`** was touched in commit `250d822` for locale-related fixes - not re-audited here, but worth a follow-up to verify the .ics summary text is locale-aware.
